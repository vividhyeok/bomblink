# Current Implemented Rules

This document describes the rules currently implemented in this prototype. It is not a claim about the original Bomb Link rules. Use this as a checklist for comparing the prototype against the real game.

## Screen And Board

- Logical canvas resolution is `240 x 320`.
- The board is `7 columns x 8 rows`.
- Board cell size is `28 px`.
- Row index starts at the top:
  - `row 0` is the top row and current death line.
  - `row 7` is the bottom row.
- At game start, only the bottom 4 rows are filled:
  - Filled rows: `row 4`, `row 5`, `row 6`, `row 7`.
  - Empty rows: `row 0`, `row 1`, `row 2`, `row 3`.

## Startup Flow

- The game starts in `banner` phase.
- `BOMB LINK` is shown for about `0.9 s`.
- `LEVEL 00` is shown until about `1.45 s`.
- Then the game enters `ready` phase.
- `READY` is shown for about `0.8 s`.
- `START` is shown until about `1.15 s`.
- After that, gameplay begins in `idle` phase.

## Controls

Default controls:

- Arrow keys / WASD: move cursor one cell.
- Space / Enter: rotate selected bomb clockwise.
- Z / Backspace: rotate selected bomb counter-clockwise.
- M: mute/unmute sound.
- Escape: pause/resume, except during banner, ready, result, level clear, or game over.
- R: restart.

Debug mode controls:

- Debug mode is enabled with `?debug=1` in the URL.
- F: manually fire the current row if that row has at least one bomb.
- H: cycle HUD label text.
- Debug HUD shows pressure countdown, flame countdown, and next side text.

## Bomb Model

Each bomb has:

- `row`, `col`
- `connectors`
- visual position and target position
- state:
  - `normal`
  - `ignited`
  - `exploding`
  - `empty` exists in type definitions but is not actively used for normal board cells.

Implemented connector shapes:

- left + right
- up + down
- up + right
- right + down
- down + left
- left + up
- up + right + down + left

There are no one-ended or three-way fuse bombs in the current implementation.

## Bomb Generation

- Bombs are generated by a deterministic pseudo-random seed.
- The seed resets on restart.
- Generated bombs choose one of the implemented connector shapes.
- Initial board generation fills only the bottom 4 rows.
- Pressure rows later generate a full new bottom row.

## Rotation

- Rotating a bomb rotates all its connectors by 90 degrees.
- Clockwise rotation maps:
  - up -> right
  - right -> down
  - down -> left
  - left -> up
- Counter-clockwise rotation maps the opposite way.
- A bomb can only be rotated when its state is `normal`.
- The game enters `rotating` phase for about `0.08 s`, then returns to `idle`.
- Input is still accepted during `rotating`.

## Connection Rule

Two adjacent bombs are connected only if both have matching facing connectors.

Example:

- Bomb A is left of Bomb B.
- A must have `right`.
- B must have `left`.
- If both are true, they are connected.

Connection search checks directions in this order:

1. up
2. right
3. down
4. left

Visually, a red/white fuse bridge is drawn only for adjacent bombs that satisfy this matching connector rule. Bombs that merely face the same direction are not connected.

## FLAMES Rule

- The game starts with `FLAMES 100`.
- `FLAMES` means remaining flame opportunities.
- `FLAMES` is not the number of bombs left to explode.
- Every flame event that starts consumes 1 flame:

```text
flamesRemaining -= 1
```

- Exploding bombs does not directly reduce `FLAMES`.
- Exploding bombs affects score, combo, best combo, total exploded count, and level.
- If `FLAMES` reaches 0, the game continues through the current flame/chain resolution, then shows a result screen.

## Flame Scheduling

- First automatic flame occurs about `12 s` after `START`.
- After that, automatic flames are scheduled by:

```text
13.5 - difficulty * 5 seconds
```

- At minimum difficulty this is about `13.5 s`.
- At maximum difficulty this is about `8.5 s`.
- The flame side alternates after each scheduled flame:
  - left side
  - right side
  - left side
  - right side
  - ...
- The next flame is previewed with a small marker above the board.
- In normal HUD, exact debug timing and side text are hidden.
- During the final `4 s` before fire, the top fire indicator gets a stronger warning highlight and `FIRE` label.

## Flame Descent And Contact

Automatic flames do not target a specific row.

- A flame descends vertically from above the board.
- Each board row is scanned once as the flame head descends.
- The flame may pass rows without igniting anything.
- A bomb can ignite only if it has an `up` connector.
- That `up` fuse must be exposed to the outside: no bomb may exist above it in the same column.
- If several exposed `up` fuses are at the same row, the internal left/right side value only decides scan order:
  - `left`: scan columns left to right.
  - `right`: scan columns right to left.

## Flame Movement

- A flame descends vertically from above the board.
- Flame travel duration is:

```text
3.1 - difficulty * 0.9 seconds
```

- At minimum difficulty this is about `3.1 s`.
- At maximum difficulty this is about `2.2 s`.

## Flame Ignition Rule

When a descending flame scans a row:

- If there is no bomb, nothing happens.
- If the bomb is not `normal`, nothing happens.
- A candidate bomb must have an `up` connector.
- There must be no bomb above that candidate in the same column.
- If the connector matches, that bomb ignites and the flame event ends.

This means a flame can pass over bombs without igniting them if their top fuse does not point outward or is blocked by another bomb above.

## Chain Search

When a bomb ignites:

- The game runs BFS from the ignited bomb.
- BFS uses the matching connector rule.
- Every connected bomb becomes part of the chain.
- Each chain node stores:
  - bomb
  - parent bomb
  - depth
  - BFS order

The first bomb in the chain is always the bomb directly touched by the flame.

## Chain Explosion Timing

After a chain starts:

- The first bomb is set to `ignited`.
- There is an initial delay of about `0.14 s`.
- Then bombs explode in BFS order.
- If the chain has 8 or more bombs, explosion interval is about `0.13 s`.
- Otherwise explosion interval is about `0.17 s`.
- When a bomb explodes, its child bombs in the BFS tree are set to `ignited`.
- Fuse burn segments are drawn from the exploding parent to each ignited child.

The current effect is not a precise per-connector path animation. It draws a straight segment between cell centers.

## Explosion Effects

When a bomb explodes:

- Bomb state becomes `exploding`.
- A flash is spawned at the bomb's visual position.
- Fire/smoke particles are spawned.
- Explosion sound plays.
- For chains of 8 or more bombs, particle count is higher.
- Screen shake only happens for chains of 4 or more bombs:
  - 4-5 bombs: weak shake
  - 6-9 bombs: medium shake
  - 10+ bombs: stronger short shake

After all bombs in the chain have been processed:

- The game enters `exploding`.
- After about `0.16 s`, exploded bombs are removed.
- Gravity is applied.
- The game enters `falling`.

## Gravity

Gravity is column-based.

After explosions:

- Remaining bombs in each column fall downward.
- Empty cells stay empty.
- The board is not immediately refilled from the top.
- No new bombs are spawned by gravity itself.

Movement speed is based on:

```text
9 * cellSize pixels per second
```

With the current `28 px` cell size, this is about `252 px/s`.

## Pressure Rows

Current pressure mode is `riseFromBottom`.

- First pressure row occurs about `18 s` after `START`.
- Later pressure rows are scheduled by:

```text
15 - difficulty * 5 seconds
```

- At minimum difficulty this is about `15 s`.
- At maximum difficulty this is about `10 s`.

When a pressure row happens:

- Every board row shifts up by 1.
- A full new row of bombs is generated at the bottom.
- The new bottom-row bombs visually enter from below.
- The game enters `spawning`, then waits for movement to settle.

The pressure mode is not yet exposed as a runtime option. Other possible modes to compare later are `fallFromTop` and `none`.

## Game Over And Result

Death-line game over:

- Death line is `row 0`.
- If `row 0` contains any bomb when a pressure row is about to be added, game over triggers.
- If a pressure row pushes bombs into `row 0`, game over is marked pending.
- After movement settles, pending game over triggers.
- Game over message is `GAME OVER`.

Flames result:

- If `FLAMES` reaches 0, the game shows `RESULT`.
- Result screen shows:
  - final level
  - final score
  - best combo
  - total bombs exploded

`LEVEL CLEAR` still exists as a phase type but is no longer the normal ending for `FLAMES 0`.

## Score

Score is awarded when a chain starts.

Formula:

```text
score += explodedCount * 10 + bonus
```

Bonus:

- 1-3 bombs: no bonus
- 4-5 bombs: +35
- 6-9 bombs: +90
- 10+ bombs: +200

The current combo value is the size of the latest chain.
Best combo stores the largest chain size during the run.

## Level

Level is score-based:

```text
level = floor(score / 500)
```

Level starts at 0 and rises as score rises.

## Difficulty Curve

Difficulty is:

```text
min(1, timePressure + scorePressure + levelPressure)
```

Where:

```text
timePressure = max(0, gameplayTime - 18) / 100
scorePressure = score / 3400
levelPressure = level * 0.08
```

Difficulty affects:

- automatic flame delay
- flame travel duration
- pressure-row delay

## Phases

Implemented phases:

- `banner`
- `ready`
- `idle`
- `rotating`
- `flamePassing`
- `fuseBurning`
- `exploding`
- `falling`
- `spawning`
- `levelClear`
- `result`
- `gameOver`
- `paused`

Gameplay time increases only during:

- `idle`
- `rotating`
- `flamePassing`
- `fuseBurning`
- `exploding`
- `falling`
- `spawning`

Gameplay time does not increase during:

- `banner`
- `ready`
- `paused`
- `levelClear`
- `result`
- `gameOver`

## HUD

Normal HUD shows:

- `LV 00`
- `FLAMES 100`
- `SCORE 00000`
- `COMBO ##` only after a combo exists
- `MUTE` if muted

Debug HUD, enabled with `?debug=1`, additionally shows:

- `P ##` pressure countdown
- `F ##` flame countdown
- scan-order/flame timing debug text

The target label can be cycled in debug mode through:

- `FIRE`
- `FLAMES`
- `TOTAL`
- `ATTACK`
- `LEFT`

This changes only the text label, not the rules.

## Known Current Limitations

- Only `flames100` is effectively active.
- `Endless` exists as a type but is not exposed as a playable mode.
- There is no next-level intermission.
- Bombs do not have one-ended or three-way fuse shapes.
- Fuse burn visuals still use simplified line segments, though normal connected fuses are now explicitly drawn as adjacent bridges.
- Manual `F` side-flame start exists only in debug mode and may not match the original game.
- The next flame side is previewed in this prototype; whether the original previewed it this way is unknown.
- Pressure currently pushes from the bottom upward. If the original used falling pieces from above or another pressure model, this differs.
